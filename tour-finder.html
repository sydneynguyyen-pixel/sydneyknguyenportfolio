Background

The idea came from watching someone close to me navigate this exact problem — they work in live music, coordinating tours and production for touring artists. Tour manager and production coordinator roles are almost never publicly listed. They're filled through personal networks, word of mouth, or someone in the industry noticing an artist is about to tour and reaching out first.

I watched how they actually found work: manually researching artists, guessing who might be gearing up for a tour, checking if they already had representation, cold-emailing when they spotted an opening. It worked — but only for whichever artists happened to come to mind, and it didn't scale.

Concept

Tour Finder is a discovery tool, not a job board.

Instead of listing open positions — which mostly don't exist in this industry — it surfaces artists showing real signs of touring soon: a fresh release with no tour booked yet, a comeback after time away, growing show frequency. It ranks them by likely fit for someone with specific touring experience and scale.

The core idea: opportunity in this kind of work is often invisible until it's too late.

By the time a tour is publicly announced, someone's usually already been hired to run it. This tool tries to surface the window before that happens.

Who This Is For

The same early-tour signal that helps one person find their next gig is a qualified-lead pipeline for any company that sells into tours.

Independent crew

TMs & production, gig to gig

Travel agencies

Flights, buses, hotels for touring parties

Staffing companies

Placing TMs, coordinators, crew

Agents & vendors

Spotting rising acts early

Business Opportunity

One prediction, many buyers — the insight that a tour is coming before it's announced is worth money to everyone who sells into tours.

Timing is the edge

By the time a tour is public, the roles and vendors are locked. Everyone who services a tour faces the same closing window the crew does; the early signal gets them in first.

Measured in lead time won

The metric that matters isn't the fit score, it's how many weeks ahead of a public announcement an act gets flagged. One landed account off an early lead covers a year of the tool.

Leads become a pipeline

The personal log becomes a team funnel: surfaced → contacted → responded → signed.

How it could grow

Free personal tier, paid per-seat for agencies, data/API licensing for vendors. Beachhead: entertainment travel agencies.

Use Cases

How it fits into the week — from first signal to a shortlist worth emailing.

Discover

A weekly-refreshed feed surfaces artists showing early signs of touring.

Research

Each lead's tour history, venue scale, and contacts in one view — no digging.

Track

Log the artists you've actually worked with to sharpen future recommendations.

Filter

Dismiss wrong-scale, wrong-genre, or already-managed acts to cut the noise.

Key Features

Scored leads

A 0–100 fit score built from five factors: touring history, venue scale match, how reachable an artist's team is, timing (a fresh release with no tour booked is the strongest signal), and momentum.

Full artist detail pages

Tour history, recent releases, contact info, and a plain-language breakdown of why an artist scored the way it did — not just a bare number.

Personal artist log

A record of who you've actually worked with, separate from suggested leads, which doubles as calibration data for the scoring system.

Adjustable genre preferences

Drag genres up or down to reorder — scoring nudges toward your current priorities in real time, without excluding anything else.

New Tour Detected

An unfiltered feed of confirmed tours already on sale but not yet underway, soonest first — pulled straight from ticketing data. Tracked artists surface immediately; anyone outside your roster needs a real multi-city run to qualify.

Automated + on-demand refresh

New leads arrive automatically every week — or trigger a check yourself anytime and watch results roll in.

How It Was Built

Stack: React 19 + Vite (dashboard), Node.js (automation pipeline), GitHub Actions (weekly cron), Netlify (hosting + serverless functions). Built solo, paired with Claude Code — the whole thing went from empty repo to a live, running pipeline in about 2 days.

Data sources

All free, public data — no paid API required.
  • Setlist.fm, MusicBrainz, TheAudioDB, Deezer, Wikidata, Last.fm, Discogs, and RSS for release, tour, and genre data
  • Ticketmaster + JamBase for confirmed on-sale dates and venues
  • Originally scoped around a major streaming platform's API — restricted mid-build, so the project moved to this multi-source setup instead

Discovery mechanism

An artist-similarity graph instead of a fixed watchlist.
  • Starts from artists you've actually worked with, traverses to sonically-adjacent ones you haven't
  • Filters out anyone already known, scores the rest
  • Personalizes automatically as you log more experience

Scoring

A weighted algorithm across the five factors behind the fit score above.
  • Timing carries the most weight — a fresh release with no tour booked yet is the strongest signal
  • Distinguishes a genuine comeback from a false signal (a remix drop after years of inactivity), weighting original material higher

Architecture

No database — GitHub itself is the backend.
  • A weekly GitHub Actions workflow runs the pipeline and commits results straight to the repo
  • The dashboard is a static React app reading that data over the GitHub raw URL — no server to keep running
  • Local edits sync back through a Netlify function, with reconciliation logic to prevent stale overwrites

Building the tool made the original problem more obvious, not less — opportunity in this industry really is just a timing window, and now the pipeline proves it every week.

Live and in active use — for one person, for now. The weekly scan runs on schedule, feeding a growing roster of tracked artists. The concept holds up in practice, which was really the question this build was answering.

The hardest problem wasn't the scoring — it was trusting a browser tab.

Keeping the dashboard's local edits in sync with the backend without silently losing data turned out to be the real challenge. Early on, a returning visit could overwrite newer backend fields with a browser's stale local copy. Fixing it meant reconciling local and backend state on every visit instead of just trusting whichever one wrote last.

Continuing: wiring genre preferences into the same sync path artist-logging already uses · tuning scoring weights as more real leads come in · opening this up beyond one user with accounts and sign-in · exploring paywalled and business-tier APIs for data depth I can't reach today.
